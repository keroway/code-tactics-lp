# 技術スタックの推奨と選定理由

> ステータス: **決定済み**。確定事項の決定記録は [decisions.md](./decisions.md)。
> 本ファイルは選定理由と代替案比較の資料として残す。

## 結論 (推奨)

紹介ページ (LP) は **コンテンツ中心の静的サイト** であり、リッチな双方向 UI もログイン
も不要。よって「速い・軽い・SEO に強い・運用が楽」を満たす構成を推奨する。

- **フレームワーク: [Astro](https://astro.build/)**
- **スタイリング: [Tailwind CSS](https://tailwindcss.com/)**
- **ホスティング: GitHub Pages または Vercel**

## なぜ Astro か

- LP のような **静的コンテンツ主体のサイトに最適化** されており、デフォルトでクライアント
  JS をほとんど出力しない → 表示が速く、SEO・Core Web Vitals に有利。
- Markdown / MDX でコンテンツを書けるので、文章主体の紹介ページと相性が良い。
- 必要な箇所だけ React/Svelte/Vue コンポーネントを「島 (Islands)」として埋め込める →
  将来インタラクティブなデモを足したくなっても拡張可能。
- 静的出力なので GitHub Pages / Vercel / Netlify どこへでも置ける。

## 代替案の比較

| 候補 | 向き / 強み | 懸念 |
| --- | --- | --- |
| **Astro** ✅ | 静的 LP に最適、JS 最小、Markdown 親和、島で部分的に動的化可 | LP 用途では特になし |
| SvelteKit | **本体 (Svelte 5 + Vite) と技術を揃えられる**、学習資産を共有 | LP には機能過多気味、静的化に設定が要る |
| Next.js | エコシステムが巨大、Vercel と好相性 | LP には重め、デフォルトで JS を多く吐く |
| 素の HTML/CSS + Vite | 最小構成・依存が少ない | コンポーネント化・コンテンツ管理が手作業になりやすい |
| VitePress / Docusaurus | ドキュメントサイト向き | マーケティング LP には型が合いにくい |

**本体とのスタック統一を重視するなら SvelteKit** が次点。チームの慣れや今後デモを本体
由来のコンポーネントで作りたいかで選ぶとよい。

## ホスティングの選択肢

- **GitHub Pages** — 本体 web 版が `keroway.github.io/code-tactics/` で配信されているため
  運用を揃えやすい。`base` パス設定に注意。
- **Vercel** — プレビューデプロイ・独自ドメインが簡単。LP を別ドメイン/サブドメインで
  出したい場合に有力。

## 決め方の指針

1. **LP 単体で完結させたい / とにかく速く軽く** → **Astro**
2. **本体と技術スタックを統一したい** → **SvelteKit**
3. いずれもホスティングは GitHub Pages か Vercel から選ぶ。
